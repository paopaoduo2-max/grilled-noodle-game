## 🎯 架构说明

### 工作原理

#### 1. 多Camera切换
```
┌─────────────────────┐
│   GameScene         │
│                     │
│  ┌──────────────┐   │
│  │ MenuCamera   │◄──┼─── 默认启用
│  └──────────────┘   │
│  ┌──────────────┐   │
│  │PrepareCamera │   │
│  └──────────────┘   │
│  ┌──────────────┐   │
│  │CookingCamera │   │
│  └──────────────┘   │
│  ┌──────────────┐   │
│  │ResultCamera  │   │
│  └──────────────┘   │
└─────────────────────┘
```

每次切换时：
1. 禁用所有Camera
2. 隐藏所有Canvas
3. 启用目标Camera
4. 显示目标Canvas

#### 2. 数据流
```
用户操作
   ↓
MainMenuUI.onStartGame()
   ↓
GameManager.startLevel()
   ↓
GameManager.enterPreparePhase()
   ↓
CameraManager.switchToPrepare()
   ↓
PrepareCamera启用 + PrepareCanvas显示
```

### 优势对比

| 特性 | 多场景架构 | 单场景多Camera架构 |
|------|-----------|-----------------|
| 场景切换速度 | 慢（加载/卸载） | 快（瞬时） |
| 内存使用 | 低（按需加载） | 高（一次加载） |
| 数据传递 | 复杂 | 简单 |
| 开发调试 | 需要切换场景 | 统一场景 |
| 向后兼容 | N/A | ✅ 支持回退 |

## ❓ 常见问题

### Q1：我必须使用单场景架构吗？
**A：不必须。** 旧场景已经修复，可以继续使用。单场景架构是优化选项。

### Q2：单场景架构会影响性能吗？
**A：两方面影响：**
- ✅ 正面：切换速度快，无加载延迟
- ⚠️ 负面：初始内存占用稍高

对于你的项目规模，影响可以忽略。

### Q3：UI复制会丢失数据吗？
**A：不会。** 但需要重新绑定：
- 节点引用
- 组件属性
建议复制后逐项检查。

### Q4：能同时保留两种架构吗？
**A：可以！** GameManager已支持向后兼容，可以随时切换。
