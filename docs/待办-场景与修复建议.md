## 当前已观察到的场景结构要点
- `MainMenu.scene` 下存在 `GameManager` 节点挂载了 `GameManager/InventoryManager/GameProgressManager`，同时还有独立的 `InventoryManager` 节点，存在单例冲突风险。
- `Level1CookingScene.scene` 的 Canvas 同时挂了 `CookingPhaseUI` 与 `CookingControllerV2`，存在“双入口逻辑”风险。
- `ShopScene.scene` 主要由 `ShopController` 驱动，`ProcessingScene.scene` 由 `ProcessingControllerV3` 驱动，`SettlementScene.scene` 由 `SettlementController` 驱动。

## 主要问题与风险（待处理）
1. **流程分裂**
   - `GameManager`（Prepare → Cooking → Result）与 `LevelManager`（Shop → Processing → Cooking → Settlement）并存。
   - 风险：关卡/金币/解锁/存档状态不一致、场景跳转逻辑混乱。

2. **单例重复挂载**
   - 例如 `MainMenu.scene` 中 `InventoryManager` 可能被多处挂载。
   - 风险：实例互相覆盖，初始化/存档重复执行。

3. **数据源重复**
   - 关卡/流程配置分散在 `assets/Scripts/Data/GameConfig.ts`、`assets/Scripts/Data/ShopData.ts`、`assets/Scripts/Manager/GameProgressManager.ts`。
   - 风险：配置修改易遗漏，数值不一致。

4. **音频系统未落地**
   - `assets/Resources/Audio`、`assets/Audio` 目录为空。
   - `AudioManager` 未见挂载，`SettingsManager` 未实际应用音量。

## 修复方向（建议）
### A. 流程与架构
- **确定唯一主流程**：建议二选一并统一入口，避免双流程。
  - 方案1：Shop → Processing → Cooking → Settlement
  - 方案2：Prepare → Cooking → Result
- **统一入口管理器**：主菜单只调用一个管理器（`GameManager` 或 `LevelManager` 其一）。
- **建立 AppRoot/Bootstrap 节点**：集中挂载 `GameProgressManager/InventoryManager/AudioManager`。

### B. 场景组件清理
- 同一场景避免挂载两套“流程入口类”（如 `CookingPhaseUI` + `CookingControllerV2`）。
- 将“数据控制”与“表现UI”拆开：UI脚本只读状态，不直接切场景。

### C. 数据配置归一
- 关卡/商品/流程配置统一为单一数据源（建议 JSON 化或集中到一个 TS 配置）。
- 保留旧结构一段时间，通过兼容层过渡，避免一次性破坏。

### D. 音效/配音管线
- 资源规范：`assets/Resources/Audio/BGM`、`assets/Resources/Audio/SFX/UI`、`assets/Resources/Audio/SFX/Cooking`、`assets/Resources/Audio/VO`
- `AudioConfig` 只保留路径与枚举 Key，事件/玩法配置只引用 Key。
- `AudioManager` 统一管理 Master/BGM/SFX/VO 音量，`SettingsManager` 应用设置。

## 特殊事件系统（新方案草案，优先1-3关）
> 目标：替换/接管“随机事件”玩法，但不破坏 `CookingControllerV2` 的手机/顾客/打包/时间等通用功能；事件不允许“补货/购买食材”，仅允许金币/热度/客流与少量主食材增减。

### 事件文案可手写覆写（已接入）
> 目的：特殊事件的剧情/发送者/选项文案需要频繁改稿测试，因此抽离到资源文件，后期你直接手写即可，游戏弹窗会按覆写显示。

- 覆写表：`assets/Resources/SpecialEvents/special_event_overrides.json`
  - 以 `eventId` 为 key（如 `L1_M01 / L2_E07 / L3_N03 / L4_G01`）
  - 可选字段：`description`、`sender`、`optionA`、`optionB`（没写的字段继续用模板默认值/兜底）
- 读取/合并逻辑：`assets/Scripts/Game/SpecialEvents/SpecialEventTextOverrides.ts`
- 使用位置：`assets/Scripts/Game/SpecialEvents/SpecialEventSystem.ts`（弹窗展示前 apply）
- 说明文档：`docs/特殊事件文案覆写指南.md`
- 注意：Creator 的 resources 可能有缓存；修改 JSON 后如未立刻生效，优先“关闭 Preview 再重新 Preview”。

### 核心约束
- **不继续扩写 V2 事件系统代码**：新做 `SpecialEventSystem`（独立目录/脚本），不往 `RandomEventSystemV2/EventManagerV2/EventSystemIntegration` 继续堆逻辑。
- **劳动型判定统一**：事件任务进度只按“完成正常顾客订单并交付收款”计数（完成N单）。
- **库存增减只动主食材**：不动调料、不出现购买/补货；主食材变动以“惩罚/风险”为主，奖励为辅且幅度小。
- **事件当日不重复**：按 `templateId` 去重；并支持 `cooldownDays` 降低隔天重复体感。

### 奖励结构（以金币/热度为主）
- 主奖励：金币、热度、次日客流（可选，幅度小，持续1天）。
- 次要奖励：小游戏“免费入场券/硬币/打折券”（见下文），用于引导玩家去玩对应小游戏。
- 主食材变动：通常为失败惩罚 `-1~-2`；极少数“大奖”允许 `+1`。

### 券系统（跨天保留，3天内有效）
> 券不属于食材库存（不走 `InventoryManager` 食材计数），单独存储；每张券记录 `issuedDay/expireDay`，超过 `expireDay` 自动失效。

- **Level1 刮刮乐免费票**：铜/银/金（对应刮刮乐三档）。
- **Level2 扭蛋硬币**：普通/高级/豪华（对应扭蛋机三档）。
- **Level3 赌石店打折券**：8折/5折/1折（三档折扣）。
- **Level4/5**：先保留券框架与占位类型，待玩法确定后补充。

### Level3 赌石打折券规则（已确认）
- 折扣档位：**8折/5折/1折**（倍率 0.8 / 0.5 / 0.1），**3天内有效**。
- 使用方式：进入赌石店后，本次**第一单**全程生效（对“买石 + 开窗 + 切割”全部打折），**离开赌石店即失效**。
- “第一单”定义建议：以“本次进入赌石店后购买的第1块原石”为准；折扣仅作用于该原石的购买/开窗/切割，后续原石不再享受折扣。

### 模板库扩容与降重复策略（1-3关先做满）
- 模板分三层组合，提升事件总量但保持可维护：
  1) **机制模板（Mechanic/Archetype）**：定义任务形式、结算结构、券类型。
  2) **文案皮肤（Skin）**：同机制多套发起人/标题/描述/按钮文案，避免“看起来重复”。
  3) **参数化（Params）**：N单/时限/奖励区间/券档位概率按“难度”动态生成。
- 每天每时段触发1次事件；同天按 `templateId` 严格去重。
- 难度与券档位：难度越高，高档券概率越高（尤其是赌石 1折券保持极稀有）。
